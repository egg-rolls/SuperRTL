# SuperRTL 能力边界与设计决策

本文档记录 SuperRTL 的能力边界和关键设计决策，避免重复讨论。

---

## FPGA 开发流程分工

### 完整 FPGA 开发流程

```
Step 1: RTL 代码编写
    ↓
Step 2: 仿真验证
    ↓
Step 3: 综合
    ↓
Step 4: 约束文件 (XDC/SDC)
    ↓
Step 5: 实现 (布局布线)
    ↓
Step 6: Bitstream 生成
    ↓
Step 7: 烧录
```

### SuperRTL 负责的部分

| 步骤 | 能力 | 说明 |
|------|------|------|
| Step 1: RTL 代码编写 | ✅ 核心能力 | Skills + Templates + 代码生成 |
| Step 2: 仿真验证 | ✅ 核心能力 | 编译 + 仿真 + 波形查看 |
| Step 3: 综合检查 | ⚠️ 部分能力 | Yosys 综合检查（非完整综合） |

### 用户/Vivado 负责的部分

| 步骤 | 原因 |
|------|------|
| Step 4: 约束文件 | 本质是查手册，5 分钟手动完成，无法通用化 |
| Step 5: 实现 | 布局布线是专有算法，依赖 Vivado/Quartus |
| Step 6: Bitstream | 需要完整工具链，非开源 |
| Step 7: 烧录 | 依赖硬件连接和厂商工具 |

### 决策依据

**为什么不做约束文件生成 (XDC)：**

1. **投入产出比低**：XDC 本质是引脚查表，用户看板卡手册 5 分钟就能写完
2. **无法通用化**：引脚映射因项目而异，预置模板只能覆盖时钟等固定引脚
3. **用户习惯**：Vivado GUI 本身有引脚分配界面，比生成文件更直观
4. **自动化程度有限**：时序约束需要理解设计的时钟域和关键路径，AI 难以准确推断

**为什么不做 Bitstream 生成：**

1. **工具链依赖**：需要完整 Vivado/Quartus，非开源
2. **实现过程封闭**：布局布线是厂商专有算法
3. **时序收敛**：需要迭代优化，无法一次性生成
4. **物理约束**：与具体 FPGA 芯片相关

## 未来扩展方向

### 可以做的

| 方向 | 优先级 | 说明 |
|------|--------|------|
| 更多 Skills | 高 | SPI、I2C、AXI、DDR 等 |
| 更好的 Testbench | 高 | 自检查、覆盖率驱动 |
| 代码审查 | 中 | 可综合性检查、时序分析 |
| 模块化设计 | 中 | IP 核库、参数化模块 |
| 形式验证 | 低 | 集成 SymbiYosys |

### 不做的

| 方向 | 原因 |
|------|------|
| 约束文件生成 | 投入产出比低，手动更快 |
| Bitstream 生成 | 依赖厂商工具链 |
| 布局布线 | 专有算法，非开源 |
| 硬件烧录 | 依赖物理连接 |
